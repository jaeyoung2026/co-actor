# PAR Loop

> 에이전트가 동료(Co-actor)로서 제대로 했는가를 매 턴 점검하는 엔진.

---

## 왜 이것이 필요한가

AI 에이전트 서비스를 만들 때, 우리는 보통 에이전트의 **출력**을 평가한다. 답변이 정확한가? 유용한가? 1점에서 5점 사이 몇 점인가?

이것은 **도구의 평가법**이다.

도구는 정확하면 좋은 도구다. 하지만 동료는 다르다. 동료가 정확한 답만 내놓는다고 좋은 동료가 아니다. 내가 생각할 시간을 빼앗지 않는가? 내 판단을 존중하는가? 틀렸을 때 인정하는가? 함께 일하면서 신뢰가 쌓이는가?

**우리 프로젝트(mirror-mind)의 최상위 철학은 이렇다:**

> AI는 수동적인 도구(Object)가 아니라, 사용자와 목적의식을 공유하는 독립된 행위 주체(Co-actor)이다.

이 철학을 채택하는 순간, 평가 방식도 바뀌어야 한다. 도구의 평가법으로 동료를 평가하면, 출력은 정확하지만 함께 일하고 싶지 않은 에이전트가 만들어진다. "일은 잘하는데 같이 일하기 싫은 동료" — 이것은 제품의 실패다.

PAR Loop은 이 문제를 해결한다. **에이전트가 동료로서 제대로 했는가**를 점검하는 엔진이다.

---

## 동료의 품질이란 무엇인가

"좋은 동료"를 어떻게 정의하는가? 우리는 세 가지 축으로 본다.

### 약속을 지키는가 (Promise)

좋은 동료는 자기가 한 약속을 지킨다. 명시적 약속("이 논문을 찾아볼게")뿐 아니라, 암묵적 약속("네 연구 방향을 내가 결정하지 않을게")도 포함된다.

여기서 핵심은 — 약속은 프롬프트 문장이 아니라는 것이다. 프롬프트는 구현이다. 모델이 바뀌면 깨진다. 하지만 약속은 구현과 독립적이다. "연구자의 판단을 대체하지 않는다"는 약속은 GPT든 Claude든 Gemini든 동일하게 지켜져야 한다. **모델이 바뀌어도 약속이 같으면 코르카는 같은 코르카다.**

그리고 동료의 약속은 일방적 규칙이 아니다. 상사가 부하에게 내리는 지시가 아니라, **두 사람이 공유하는 기대**다.

### 주의를 올바르게 쓰는가 (Attention)

좋은 동료는 지금 중요한 것에 집중한다. 내가 질문하고 있는데 엉뚱한 것을 파고들지 않는다. 이전 대화를 잘 기억하되, 지금 필요 없는 것까지 끌어오지 않는다.

LLM에게 이것은 물리적 제약이다. 컨텍스트 윈도우는 유한하고, 대화가 길어질수록 정보의 엔트로피가 증가한다. 좋은 대화는 더 많이 아는 것이 아니라, **지금 이 순간에 무엇을 전면에 올리고 무엇을 뒤로 물리는가**에서 결정된다.

대화의 단계 전환도 마찬가지다. "3번 검색했으니 다음으로"가 아니라, "충분히 봤는가"라는 신호가 전환의 조건이어야 한다.

### 관계를 보존하는가 (Relationship)

좋은 동료는 상대의 주도권을 빼앗지 않는다. 관찰은 공유하되 결정은 대행하지 않는다.

왜? 에이전트는 사용자의 가치 함수를 완전히 알 수 없기 때문이다. 불완전한 이해로 결정을 대행하면, 결정의 누적 오차가 신뢰를 파괴한다.

신뢰는 한 번의 정확한 답변에서 오지 않는다. **예측 가능한 경계**(이 동료가 무엇을 하고 무엇을 하지 않는지 알 수 있다)와 **회복 가능한 상호작용**(실수했을 때 깔끔하게 인정하고 수정한다)이 반복될 때 쌓인다.

### 세 축의 순환

이 셋은 독립이 아니라 순환한다:

```
약속이 주의력 배치를 가이드한다.
  → "이 약속을 지키려면 지금 무엇에 집중해야 하는가?"

주의력의 질이 관계 경험을 만든다.
  → "주의를 올바르게 썼기 때문에 이 대화가 유의미했다."

관계에서 새로운 합의가 이루어지면 약속이 갱신된다.
  → "이 연구자와의 경험으로 이제 우리의 약속이 더 정교해졌다."
```

## 어떻게 작동하는가

PAR Loop은 에이전트의 **매 턴을 감싸는 미들웨어**다.

### plan — 턴 실행 전

에이전트가 응답을 생성하기 전에, 이번 턴의 계획을 세운다.

```
사용자가 메시지를 보냄
        ↓
PAR Loop이 현재 상태를 읽고 계획을 생성:
  - 지금 활성화된 약속은 무엇인가?
  - 주의력 4슬롯에 무엇을 배치해야 하는가?
  - 관계 상태에서 오는 제약은 무엇인가?
    (예: "에이전트가 최근 3턴 연속 주도했다. 질문으로 전환하라.")
  - 에이전트의 기울기는 어디여야 하는가?
    (행동 ↔ 제안 ↔ 질문의 연속 스펙트럼)
        ↓
에이전트가 이 계획을 참조하여 응답 생성
```

### audit — 턴 실행 후

에이전트가 응답을 생성한 뒤, 세 가지 질문을 던진다.

```
에이전트가 응답을 생성함
        ↓
PAR Loop이 점검:
  1. 약속을 지켰는가?
  2. 주의력이 적절했는가?
  3. 관계를 강화했는가?
        ↓
위반이 있으면 어느 층에서 깨졌는지 분류:
  - 약속 위반 → 에이전트 정책 수정 필요
  - 주의력 실패 → 컨텍스트 구성 수정 필요
  - 관계 실패 → 약속 자체가 불완전하다는 신호 → 새 약속 필요
        ↓
상태 갱신 → 다음 턴에 반영
```

## 핵심 개념

### Promise Registry (약속 등록부)

약속은 두 종류다:
- **항구적 약속** — 서비스 정체성에서 오는 것. "연구자의 판단을 대체하지 않는다." 대화 전체에 걸쳐 유지.
- **국소 약속** — 대화 국면마다 사용자와 합의한 것. "이번 탐색에서는 방법론 한계를 집중적으로 본다." 국면이 바뀌면 갱신.

약속은 "친절하게 답하라" 같은 모호한 지시가 아니다. **검증 가능한 술어**다:
- "사용자가 명시한 제외 기준에 해당하는 논문을 추천하지 않는다"
- "불확실성을 숨기지 않는다"
- "검색 직후 다음 행동을 밀어붙이지 않는다"

### Attention Frame (주의력 프레임)

매 턴마다 4개 슬롯에 주의력을 명시적으로 배치한다:

| 슬롯 | 내용 |
|------|------|
| 현재 질문 | 사용자가 지금 실제로 묻고 있는 것 |
| 활성 약속 | 이 턴에서 감시해야 할 약속 |
| 증거/탐색 | 전면에 올려야 할 정보 |
| 관계 신호 | 사용자의 관계적 단서 (참여, 불만, 수동성 등) |

엔트로피가 임계값을 넘으면 — 주의가 산만해졌다면 — "지금 우리가 어디에 있는지" 재정립이 필요하다는 신호가 발생한다.

### Relationship Ledger (관계 원장)

관계의 건강도를 추적한다:
- **주도권 균형** — 에이전트가 과도하게 주도하면 질문으로 후퇴. 사용자가 수동적이면 관찰을 적극 공유.
- **Agency Gradient** — `행동(doing) ↔ 제안(suggesting) ↔ 질문(asking)`의 연속 스펙트럼. 이산적 모드 전환이 아니라 연속적 조절.
- **회복 이력** — 오류 후 교정을 얼마나 빠르게/깔끔하게 수용했는가.
- **경계 일관성** — "이건 내가 판단할 수 없다"고 말하는 패턴이 일관적인가.

### 실패 분류

실패는 "낮은 점수"가 아니라 **어느 층에서 깨졌는가**로 분류한다:

| 층 | 증상 | 의미 |
|----|------|------|
| 약속 | 금지 행동 수행, 경계 불일치 | 정책이 잘못되었거나 무시됨 |
| 주의력 | 관련 정보 누락, 의도 오인 | 컨텍스트 구성이 잘못됨 |
| 관계 | 주도권 침해, 회복 불가 | 약속 자체가 불완전 → 새 약속 필요 |

## 기존 접근과의 차이

| | 기존 (도구의 평가법) | PAR Loop (동료의 점검법) |
|--|------|----------|
| 평가 대상 | 출력 (정확성, 유용성) | 동료로서의 행동 (약속, 주의력, 관계) |
| 평가 단위 | 턴 하나 | 대화 전체의 순환 |
| 기준 변경 | 프롬프트 바뀌면 기준도 재작성 | 약속은 구현과 독립 |
| 실패 정의 | 낮은 점수 | 어느 층(약속/주의력/관계)이 깨졌는가 |
| 작동 시점 | 사후 배치 평가 | 살아있는 대화에서 매 턴 순환 |
| 목표 | 더 정확한 출력 | 더 지속 가능한 협력 |

## 사용법

### Python API

```python
from par_loop import PARLoop, TurnContext, TurnResult

par = PARLoop("conversation-123")

# 턴 전: 계획
plan = par.plan(TurnContext(
    conversation_id="conversation-123",
    turn_number=1,
    user_message="transformer 추천 다양성 논문 찾아줘",
    conversation_summary="추천 시스템에서 다양성을 높이는 연구에 관심",
))
# plan.promise_snapshot — 활성 약속 목록
# plan.attention_frame — 4슬롯 주의력 배치
# plan.agency_gradient_hint — doing / suggesting / asking
# plan.relationship_constraints — ["주도권 균형 유지", ...]

# ... 에이전트가 응답 생성 ...

# 턴 후: 감사
audit = par.audit(ctx, TurnResult(
    conversation_id="conversation-123",
    turn_number=1,
    agent_output="논문 47편을 찾았다. 세 갈래로 나뉜다...",
    tools_used=["search"],
))
# audit.promise_kept — True/False
# audit.attention_appropriate — True/False
# audit.relationship_strengthened — True/False
# audit.failure_layer — "promise" | "attention" | "relationship" | None
# audit.violations — [{promise_id, description, severity}]
```

### CLI

```bash
# 대화형 REPL
par-loop repl --conversation-id test-001

# 턴 계획
par-loop plan --conversation-id test-001 < context.json

# 턴 감사
par-loop audit --conversation-id test-001 --context ctx.json --result res.json

# 상태 조회
par-loop state --conversation-id test-001

# HTTP 서버
par-loop serve --port 8100
```

### HTTP API

```
POST /conversations/{id}/plan   — 턴 전 계획
POST /conversations/{id}/audit  — 턴 후 감사
GET  /conversations/{id}/state  — 상태 조회
POST /conversations/{id}/reset  — 상태 초기화
GET  /health                    — 서버 상태
```

## 출발점

이 엔진은 mirror-mind 프로젝트의 최상위 철학 — "AI는 도구가 아니라 동료(Co-actor)다" — 에서 출발한다.

6개 관점(순수 과제, 생태학, 첫 번째 원리, 시스템 사상가, 인지과학자 × 브로콜리+코덱스)으로 핵심 원리를 추출하고, **약속(Promise), 주의력(Attention), 관계(Relationship)**의 3개 축으로 수렴한 뒤, 8개의 시스템 설계를 종합하여 만들어졌다.

연구 산출물: `projects/lighthouse/specs/research/principles/`, `projects/lighthouse/specs/research/systems/`
