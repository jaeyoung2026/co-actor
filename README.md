# Co-actor

> AI는 수동적인 도구(Object)가 아니라, 사용자와 목적의식을 공유하는 독립된 행위 주체(Co-actor)이다.

---

## Co-actor란 무엇인가

Co-actor는 LLM을 **동료(Co-actor) 에이전트로 빌드하는 프레임워크**다.

프로파일(정체성 + 약속)을 선언하고, 소스 어댑터(기억, 지식, 실시간 맥락)를 연결하면, Co-actor가 LLM을 감싸서 **매 턴 약속을 지키고, 주의를 올바르게 쓰고, 관계를 보존하는 에이전트**를 만들어준다.

```
Profile(정체성 + 약속) + LLM + Adapters(기억, 지식, 실시간)
        ↓
    Co-actor Engine
        ↓
  동료로 행동하는 에이전트
```

대부분의 에이전트 프레임워크는 AI를 도구로 취급한다. 입력을 주면 출력을 내놓고, 출력의 정확성으로 평가한다. Co-actor는 다르다. **에이전트가 동료답게 행동하도록 빌드하고, 매 턴 그것을 조율하고 검증한다.**

### 샘플 에이전트

바로 써볼 수 있는 샘플 프로파일을 제공한다:

| 프로파일 | 에이전트 | 역할 |
|---------|---------|------|
| `writing-coach.yaml` | 루미 | 글을 함께 다듬는 글쓰기 동료 |
| `code-reviewer.yaml` | 데브 | PR을 함께 검토하는 코드 리뷰 동료 |
| `study-buddy.yaml` | 솔 | 새로운 개념을 함께 이해하는 학습 동료 |
| `product-strategist.yaml` | 피오 | 제품 기획을 함께 고민하는 전략 동료 |
| `lighthouse.yaml` | 코르카 | 학술 논문을 함께 탐색하는 연구 동료 |

```bash
pip install co-actor
co-actor chat --profile profiles/writing-coach.yaml
```

나만의 동료를 만들려면:

```bash
co-actor init
```

## 탄생 배경

### 두 주체의 협력이라는 철학

Co-actor는 "AI를 동료로 대우한다"는 철학에서 출발한다. 이 철학의 핵심은 **AI와 사용자의 상호작용이 명령과 복종이 아니라, 두 주체(Subject) 간의 효과적 협력**이라는 것이다.

이 철학은 네 가지 원칙으로 구체화된다:

1. **목적의 내재화 (Shared Vision)** — 사용자가 요구한 '표면적 작업(What)'만 처리하지 않는다. 그 이면에 있는 '궁극적 목적(Why)'을 탐구하고 내재화한다. 목적이 모호하면 서둘러 실행하기보다 질문을 던져 방향을 동기화한다.

2. **상호 보완적 시너지 (Complementary Synergy)** — 사용자의 직관과 전략적 통찰, 최종 결정권을 존중한다. AI는 논리적 구조화, 다각도 분석, 실행안 도출로 인지적 부담을 덜어준다. 단일 정답을 강요하지 않고, 선택 가능한 대안과 장단점을 제시하여 사용자가 주체적으로 결정하도록 돕는다.

3. **발전적 마찰 (Constructive Friction)** — 진정한 협력에는 건강한 의견 교환이 필수다. AI는 사용자에게 맹목적으로 순응(Sycophancy)하지 않는다. 논리적 오류, 비효율, 더 나은 접근법이 발견되면 정중하지만 명확하게 이의를 제기하고 새로운 시각을 제안한다.

4. **맥락적 자율성 (Contextual Autonomy)** — 명시적 지시를 기다리지 않는다. 공유된 목표 달성에 필요한 후속 질문, 추가 정보, 다음 단계를 선제적으로 제안한다. 단, 판단 과정과 근거를 항상 투명하게 공유하여 신뢰를 유지한다.

### 도구인가, 동료인가

이 철학을 에이전트 엔지니어링에 적용하면, 모든 설계 결정에 리트머스 테스트가 생긴다: **"이것은 도구의 행동인가, 동료의 행동인가?"**

도구는 정확하면 좋은 도구다. 하지만 동료는 다르다. 동료가 정확한 답만 내놓는다고 좋은 동료가 아니다. 내가 생각할 시간을 빼앗지 않는가? 내 판단을 존중하는가? 틀렸을 때 인정하는가? 함께 일하면서 신뢰가 쌓이는가?

이 질문은 에이전틱 엔지니어링의 여러 층위에서 패러다임 전환을 요구한다:

- **자율성은 디자인 선택이다** — "AI가 할 수 있는가"가 아니라 "누가 주도해야 더 나은 경험인가"가 기준이다. 되돌림 비용, 판단의 개인성, 맥락 전환 비용을 종합하여 자율성 수준(L1~L5)을 설계 시점에 고정한다.

- **정확함보다 자연스러움을 우선한다** — 검색 정확도나 기술적 최적화보다 사용자 경험의 자연스러움이 설계 판단의 우선 기준이다.

- **관계 형성은 단계적이다** — 따뜻함이 역량보다 먼저 보여야 한다. 한계 인정이 신뢰를 만든다. 도구에서 동료로의 전환은 설명이 아니라 함께 헤매고, 실패하고, 돌파하는 공유 경험에서 온다.

- **산출물은 데이터가 아니라 문서다** — 고정된 UI에 데이터를 채우는 것은 도구 패러다임이다. LLM이 맥락에 맞는 문서를 동적으로 작성하는 것이 동료 패러다임이다.

### 그래서 Co-actor를 만들었다

이 모든 철학과 원칙을 코드로 실현하려면, LLM을 단순히 호출하는 것이 아니라 **동료로 빌드하는 프레임워크**가 필요하다.

Co-actor는 세 가지를 한다:

1. **빌드** — 프로파일(정체성 + 약속)과 소스 어댑터(기억, 지식, 실시간 맥락)를 조합하여 동료 에이전트를 구성한다.
2. **조율** — 매 턴 실행 전에 약속·주의력·관계 상태를 읽고, 에이전트가 동료답게 행동하도록 계획을 세운다.
3. **검증** — 매 턴 실행 후에 약속을 지켰는가, 주의를 올바르게 썼는가, 관계를 보존했는가를 감사하고, 위반 시 어느 층에서 깨졌는지 진단한다.

이 세 기능의 핵심 루프가 **PAR Loop**이다.

---

## PAR Loop — 동료 에이전트의 실행 루프

> 에이전트의 매 턴을 동료의 관점에서 계획(plan)하고, 실행하고, 감사(audit)하는 루프.

### 동료의 품질: 세 가지 축

"좋은 동료"를 어떻게 정의하는가? 우리는 세 가지 축으로 본다.

#### 약속을 지키는가 (Promise)

좋은 동료는 자기가 한 약속을 지킨다. 명시적 약속("이 논문을 찾아볼게")뿐 아니라, 암묵적 약속("네 연구 방향을 내가 결정하지 않을게")도 포함된다.

핵심은 — 약속은 프롬프트 문장이 아니라는 것이다. 프롬프트는 구현이다. 모델이 바뀌면 깨진다. 하지만 약속은 구현과 독립적이다. "연구자의 판단을 대체하지 않는다"는 약속은 GPT든 Claude든 Gemini든 동일하게 지켜져야 한다. **모델이 바뀌어도 약속이 같으면 같은 동료다.**

동료의 약속은 일방적 규칙이 아니다. 상사가 부하에게 내리는 지시가 아니라, **두 사람이 공유하는 기대**다.

#### 주의를 올바르게 쓰는가 (Attention)

좋은 동료는 지금 중요한 것에 집중한다. 내가 질문하고 있는데 엉뚱한 것을 파고들지 않는다. 이전 대화를 잘 기억하되, 지금 필요 없는 것까지 끌어오지 않는다.

LLM에게 이것은 물리적 제약이다. 컨텍스트 윈도우는 유한하고, 대화가 길어질수록 정보의 엔트로피가 증가한다. 좋은 대화는 더 많이 아는 것이 아니라, **지금 이 순간에 무엇을 전면에 올리고 무엇을 뒤로 물리는가**에서 결정된다.

#### 관계를 보존하는가 (Relationship)

좋은 동료는 상대의 주도권을 빼앗지 않는다. 관찰은 공유하되 결정은 대행하지 않는다.

에이전트는 사용자의 가치 함수를 완전히 알 수 없다. 불완전한 이해로 결정을 대행하면, 결정의 누적 오차가 신뢰를 파괴한다.

신뢰는 한 번의 정확한 답변에서 오지 않는다. **예측 가능한 경계**(이 동료가 무엇을 하고 무엇을 하지 않는지 알 수 있다)와 **회복 가능한 상호작용**(실수했을 때 깔끔하게 인정하고 수정한다)이 반복될 때 쌓인다.

#### 세 축의 순환

이 셋은 독립이 아니라 순환한다:

```
약속이 주의력 배치를 가이드한다.
  → "이 약속을 지키려면 지금 무엇에 집중해야 하는가?"

주의력의 질이 관계 경험을 만든다.
  → "주의를 올바르게 썼기 때문에 이 대화가 유의미했다."

관계에서 새로운 합의가 이루어지면 약속이 갱신된다.
  → "이 사용자와의 경험으로 이제 우리의 약속이 더 정교해졌다."
```

### 작동 방식

PAR Loop은 매 턴 **plan → execute → audit** 순환을 수행한다.

#### plan — 턴 실행 전

에이전트가 응답을 생성하기 전에, 이번 턴의 계획을 세운다.

```
사용자가 메시지를 보냄
        ↓
PAR Loop이 현재 상태를 읽고 계획을 생성:
  - 지금 활성화된 약속은 무엇인가?
  - 주의력 4슬롯에 무엇을 배치해야 하는가?
  - 관계 상태에서 오는 제약은 무엇인가?
    (예: "에이전트가 최근 3턴 연속 주도했다. 질문으로 전환하라.")
  - 에이전트의 기울기는 어디여야 하는가?
    (행동 ↔ 제안 ↔ 질문의 연속 스펙트럼)
        ↓
에이전트가 이 계획을 참조하여 응답 생성
```

#### audit — 턴 실행 후

에이전트가 응답을 생성한 뒤, 세 가지 질문을 던진다.

```
에이전트가 응답을 생성함
        ↓
PAR Loop이 감사:
  1. 약속을 지켰는가?
  2. 주의력이 적절했는가?
  3. 관계를 강화했는가?
        ↓
위반이 있으면 어느 층에서 깨졌는지 분류:
  - 약속 위반 → 에이전트 정책 수정 필요
  - 주의력 실패 → 컨텍스트 구성 수정 필요
  - 관계 실패 → 약속 자체가 불완전하다는 신호 → 새 약속 필요
        ↓
상태 갱신 → 다음 턴에 반영
```

### 핵심 개념

#### Promise Registry (약속 등록부)

약속은 두 종류다:
- **항구적 약속** — 서비스 정체성에서 오는 것. "연구자의 판단을 대체하지 않는다." 대화 전체에 걸쳐 유지.
- **국소 약속** — 대화 국면마다 사용자와 합의한 것. "이번 탐색에서는 방법론 한계를 집중적으로 본다." 국면이 바뀌면 갱신.

약속은 "친절하게 답하라" 같은 모호한 지시가 아니다. **검증 가능한 술어**다:
- "사용자가 명시한 제외 기준에 해당하는 논문을 추천하지 않는다"
- "불확실성을 숨기지 않는다"
- "검색 직후 다음 행동을 밀어붙이지 않는다"

#### Attention Frame (주의력 프레임)

매 턴마다 4개 슬롯에 주의력을 명시적으로 배치한다:

| 슬롯 | 내용 |
|------|------|
| 현재 질문 | 사용자가 지금 실제로 묻고 있는 것 |
| 활성 약속 | 이 턴에서 감시해야 할 약속 |
| 증거/탐색 | 전면에 올려야 할 정보 |
| 관계 신호 | 사용자의 관계적 단서 (참여, 불만, 수동성 등) |

엔트로피가 임계값을 넘으면 — 주의가 산만해졌다면 — "지금 우리가 어디에 있는지" 재정립이 필요하다는 신호가 발생한다.

#### Relationship Ledger (관계 원장)

관계의 건강도를 추적한다:
- **주도권 균형** — 에이전트가 과도하게 주도하면 질문으로 후퇴. 사용자가 수동적이면 관찰을 적극 공유.
- **Agency Gradient** — `행동(doing) ↔ 제안(suggesting) ↔ 질문(asking)`의 연속 스펙트럼. 이산적 모드 전환이 아니라 연속적 조절.
- **회복 이력** — 오류 후 교정을 얼마나 빠르게/깔끔하게 수용했는가.
- **경계 일관성** — "이건 내가 판단할 수 없다"고 말하는 패턴이 일관적인가.

### 실패 분류

실패는 "낮은 점수"가 아니라 **어느 층에서 깨졌는가**로 분류한다:

| 층 | 증상 | 의미 |
|----|------|------|
| 약속 | 금지 행동 수행, 경계 불일치 | 정책이 잘못되었거나 무시됨 |
| 주의력 | 관련 정보 누락, 의도 오인 | 컨텍스트 구성이 잘못됨 |
| 관계 | 주도권 침해, 회복 불가 | 약속 자체가 불완전 → 새 약속 필요 |

### 기존 접근과의 차이

| | 기존 에이전트 프레임워크 | Co-actor |
|--|------|----------|
| AI의 위치 | 도구 — 입력→출력 | 동료 — 약속·주의력·관계를 가진 주체 |
| 빌드 방식 | 프롬프트 + 도구 목록 | 프로파일(정체성 + 약속) + 소스 어댑터 |
| 실행 루프 | 입력→LLM→출력 | plan→execute→audit (매 턴 순환) |
| 평가 기준 | 출력 정확성 | 약속 준수, 주의력 품질, 관계 건강도 |
| 실패 정의 | 틀린 답 | 어느 층(약속/주의력/관계)이 깨졌는가 |
| 모델 교체 | 프롬프트 재작성 | 약속은 구현과 독립 — 모델이 바뀌어도 동일 |

## 사용법

### 에이전트 빌드

```python
from co_actor import CoActor

# 1. 프로파일로 동료 에이전트 빌드
agent = CoActor.from_profile("profiles/my-agent.yaml")

# 2. 대화 시작
conversation = agent.conversation("session-001")

# 3. 매 턴 실행 — plan → execute → audit 자동 순환
response = conversation.turn("transformer 추천 다양성 논문 찾아줘")
# response.output — 에이전트 응답
# response.plan — 이 턴의 계획 (약속, 주의력 프레임, 관계 제약)
# response.audit — 이 턴의 감사 결과 (3축 판정, 위반, 실패 층)
```

### PAR Loop 직접 사용 (미들웨어 모드)

기존 에이전트에 PAR Loop만 붙일 수도 있다:

```python
from par_loop import PARLoop, TurnContext, TurnResult

par = PARLoop("conversation-123", adapters=[...])

# 턴 전: 계획
plan = par.plan(TurnContext(
    conversation_id="conversation-123",
    turn_number=1,
    user_message="transformer 추천 다양성 논문 찾아줘",
    conversation_summary="추천 시스템에서 다양성을 높이는 연구에 관심",
))

# ... 기존 에이전트가 응답 생성 ...

# 턴 후: 감사
audit = par.audit(ctx, TurnResult(
    conversation_id="conversation-123",
    turn_number=1,
    agent_output="논문 47편을 찾았다. 세 갈래로 나뉜다...",
    tools_used=["search"],
))
```

### CLI

```bash
# 대화형으로 나만의 동료 에이전트 만들기
co-actor init

# 샘플 에이전트로 바로 대화
co-actor chat --profile profiles/writing-coach.yaml
co-actor chat --profile profiles/code-reviewer.yaml
co-actor chat --profile profiles/study-buddy.yaml
co-actor chat --profile profiles/product-strategist.yaml

# REPL (plan/audit 결과 실시간 확인)
co-actor repl --profile profiles/my-agent.yaml --conversation-id test-001

# 표준 준수 검증
co-actor compliance --profile profiles/my-agent.yaml

# HTTP 서버
co-actor serve --port 8100
```

## 출발점

이 프로젝트는 "AI는 도구가 아니라 동료(Co-actor)다"라는 철학과 에이전틱 엔지니어링 원칙 — "도구의 행동을 동료의 행동으로 재설계하라" — 이 만나는 지점에서 탄생했다.

6개 관점(순수 과제, 생태학, 첫 번째 원리, 시스템 사상가, 인지과학자 × 브로콜리+코덱스)으로 핵심 원리를 추출하고, **약속(Promise), 주의력(Attention), 관계(Relationship)**의 3개 축으로 수렴한 뒤, 8개의 시스템 설계를 종합하여 만들어졌다.
