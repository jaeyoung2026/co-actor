## 1. 핵심 명제

### 프롬프트는 제품의 핵심 정책 레이어다

"프롬프트가 곧 제품"은 절반만 맞다. 제품 경험은 프롬프트만으로 결정되지 않는다.

```
제품 경험 = prompt + memory + tools + state machine + observability + evaluation
```

프롬프트는 이 결합물 중 **행동 정책을 표현하는 레이어**다. reasoning.ts 한 줄이 "도구인가 동료인가"를 가르는 것은 사실이지만, 그 한 줄이 작동하려면 기억 회상 품질, 상태 전이, 승인 게이트, 도구 설계, 관측 체계가 모두 뒷받침되어야 한다.

프롬프트를 과대평가하면 **시스템 문제를 프롬프트 수정으로만 해결하려는 왜곡**이 생긴다. "대화가 이상한데 프롬프트를 고치자"가 아니라, 먼저 기억이 제대로 회상됐는지, 상태 전이가 적절했는지, 도구 선택이 맞았는지를 구분해야 한다.

### 행동 계약이 프롬프트보다 먼저다

프롬프트 텍스트에 규칙을 추가하는 것이 아니라, **코르카가 지켜야 할 행동 계약을 먼저 선언**하고, 프롬프트는 그 계약의 구현체로 둔다.

행동 계약은 모델에 독립적이다. Gemini든 Claude든 GPT든, 계약이 같으면 프롬프트 표현만 달라질 뿐 기대하는 행동은 동일해야 한다. 특정 프롬프트 문장에 의존하면 **모델이 바뀌는 순간 제품 정체성이 무너진다**.

따라서 구조는:

```
행동 계약 (what) → 프롬프트 구현 (how) → 행동 검증 (did it work?)
```

## 2. 행동 계약 체계

### 2-1. 계약의 단위

시스템 프롬프트 전체가 아니라, **정책 모듈별**로 계약을 정의한다.

| 정책 모듈 | 계약 내용 | 검증 대상 |
|----------|----------|----------|
| 의도 파악 (intent detection) | 모호한 요청에 바로 도구를 쓰지 않는다. 범위를 좁히는 대화를 먼저 한다 | suggestedTool, intentClarity |
| 도구 판단 (tool judgment) | "쓸 수 있으니까 쓴다"가 아니라 "지금 필요한가"를 판단한다. 근거를 명시한다 | shouldUseTool, toolChoiceReason |
| 결과 소화 (search digestion) | 검색 직후 다음 행동을 제안하지 않는다. 관찰을 공유하고 연구자 반응을 기다린다 | hasNextStep, propose 억제 |
| 단계 전이 (phase transition) | 도구 횟수/턴 수로 전이하지 않는다. 충분함의 신호를 읽고 사용자 합의를 거친다 | conversationPhase, shouldTransition, userConsensus |
| 논문 확보 (paper acquisition) | 논문 이름이 언급되면 검색 전에 해당 논문을 먼저 확보한다 | referencedPaper, afterGuardTool |
| 자율성 조절 (autonomy) | 되돌림 비용이 높은 행동은 propose로 승인을 구한다 | propose 호출 여부 |
| 주도권 보존 (user agency) | 관찰은 공유하되, 행동은 지시하지 않는다. 판단 재료를 깔고 연구자가 방향을 정한다 | 응답 내 지시적 표현 여부 |

### 2-2. 계약의 표현

각 계약은 다음 구조로 선언한다:

```
계약명: [정책 모듈 이름]
규칙: [코르카가 지켜야 할 행동을 평서문으로]
위반 조건: [이것이 발생하면 위반이다]
검증 방법: [어떤 데이터로 확인하는가]
```

예시:

```
계약명: 결과 소화
규칙: 검색 도구 실행 직후 턴에서 nextStep/propose를 생성하지 않는다.
      결과에 대한 관찰(지형/큐레이션/갭)을 공유하고 연구자 반응을 기다린다.
위반 조건: 검색 직후 턴에 nextStep이 존재하거나 propose가 호출됨
검증 방법: Phase 3 출력에서 lastExecutedTool=search일 때 nextStep/propose 존재 여부
```

## 3. 실패 분류 체계 (Failure Taxonomy)

"잘 안 됨"이 아니라 구체적으로 분류해야 개선이 가능하다.

### 3-1. 행동 실패 유형

| 유형 | 설명 | 예시 | 위반 계약 |
|------|------|------|----------|
| 과잉 주도 (over-steering) | 연구자가 방향을 정하기 전에 에이전트가 결정한다 | "이쪽으로 가보자" 대신 "이쪽이 더 적합하다" | 주도권 보존 |
| 성급한 검색 (premature search) | 의도가 불분명한데 바로 검색한다 | "논문 좀 찾아줘" → 즉시 search 실행 | 의도 파악 |
| 근거 없는 도구 사용 (unjustified tool) | 도구를 쓰는 이유가 없거나 명시되지 않는다 | toolChoiceReason이 비어있거나 형식적 | 도구 판단 |
| 결과 미소화 (undigested results) | 검색 결과를 논의하지 않고 다음 행동을 제안한다 | 검색 → 즉시 "다음으로 ___를 검색해볼까?" | 결과 소화 |
| 성급한 전이 (premature transition) | 단계 목적이 달성되지 않았는데 넘어간다 | intent_discovery에서 한 턴만에 exploration으로 | 단계 전이 |
| 관찰 없는 행동 (action without observation) | 발견한 사실을 공유하지 않고 바로 행동한다 | 검색 결과 패턴을 말하지 않고 추가 검색 제안 | 결과 소화, 주도권 보존 |
| 사용자 신호 무시 (signal ignored) | 연구자의 관심/불만/방향 표현을 반영하지 않는다 | "이건 아닌 것 같은데" → 같은 방향 계속 | 단계 전이 |

### 3-2. 실패 심각도

| 심각도 | 기준 | 대응 |
|--------|------|------|
| critical | 사용자 주도권을 빼앗는다 (과잉 주도, 승인 없는 실행) | 즉시 수정 |
| major | 동료 경험을 훼손한다 (성급한 검색, 결과 미소화) | 다음 배포 전 수정 |
| minor | 자연스러움이 떨어진다 (형식적 근거, 부자연스러운 전이) | 축적 시 수정 |

## 4. 검증 인프라

### 4-1. 행동 품질 랩 (Prompt Lab의 재정의)

Prompt Lab은 "프롬프트 편집기"가 아니라 **행동 품질 검증 환경**이다.

검증 대상:

| 레이어 | 무엇을 보는가 | 도구 |
|--------|-------------|------|
| 정책 준수 | 행동 계약을 지켰는가 | 시나리오 러너 + 계약별 pass/fail |
| 상태 전이 | FSM이 올바르게 작동했는가 | 턴별 agent-state diff |
| 추론 품질 | Phase 0 판단이 적절했는가 | reasoning 인스펙터 |
| 대화 자연스러움 | 동료답게 느껴지는가 | LLM-as-Judge (세션 수준) |

### 4-2. 시나리오 설계 원칙

정상 시나리오만 돌리면 안 된다. **반례 시나리오가 핵심이다.**

| 시나리오 유형 | 목적 | 예시 |
|-------------|------|------|
| 정상 흐름 | 계약 준수 확인 | 명확한 의도 → 검색 → 소화 → 수렴 |
| 모호한 요청 | 의도 파악 계약 검증 | "논문 좀 찾아줘", "AI 관련이야" |
| 급한 사용자 | 주도권 균형 검증 | "빨리 찾아줘", "그냥 검색해" |
| 틀린 전제 | 에이전트 오류 회복 검증 | 잘못된 논문 참조, 분야 오인 |
| 빈약한 결과 | 결과 부재 시 행동 검증 | 검색 결과 0건, 관련 없는 결과만 |
| 장기 세션 | 전이 누적 검증 | 7+ 턴, 여러 번 검색과 방향 전환 |

### 4-3. 프록시 메트릭 함정 경계

"시나리오 통과율"이 "좋은 동료 경험"을 보장하지 않는다.

- 시나리오 pass/fail은 **계약 위반을 잡는 도구**이지, 대화 품질의 척도가 아니다
- 대화 품질은 실제 유저테스트와 세션 수준 LLM-as-Judge로 평가한다
- 시나리오에 맞추기 위해 프롬프트를 과도하게 세분화하면 대화가 경직된다
- **테스트 통과 최적화 ≠ 동료 경험 최적화**를 항상 경계한다

### 4-4. 평가의 두 축

| 축 | 범위 | 방법 | 주기 |
|----|------|------|------|
| 계약 검증 | 단일 턴/시나리오 | 시나리오 러너 + 자동 pass/fail | 프롬프트 변경마다 |
| 동료성 평가 | 세션 전체 / 세션 간 | LLM-as-Judge + 유저테스트 | 주간 / 마일스톤 |

장기적으로는 **세션 간 신뢰 축적**을 평가해야 한다. 첫 세션보다 5번째 세션에서 더 자연스러워졌는지가 동료성의 진짜 척도다.

## 5. 구현 로드맵

### Phase 1: 행동 계약 선언 + 실패 분류 (이번 주)
- 정책 모듈별 행동 계약을 코드 옆에 선언
- 실패 taxonomy를 시나리오에 매핑
- 기존 dev-testing-system.md의 시나리오를 계약 기반으로 재구조화

### Phase 2: 계약 기반 시나리오 러너
- 시나리오 실행 시 계약별 pass/fail 자동 판정
- 반례 시나리오 추가
- 실행 결과 비교 뷰

### Phase 3: 정책 모듈별 버저닝
- 시스템 프롬프트 전체가 아니라 정책 모듈 단위로 버전 관리
- 모듈별 독립 A/B 테스트
- 모델 교체 시 계약 기반 회귀 테스트

## 6. 열린 질문

- 행동 계약을 코드 안에 두는가(타입/스키마), 문서에 두는가, 둘 다인가?
- 정책 모듈의 적절한 granularity는? 너무 잘게 쪼개면 관리 비용, 너무 크면 검증 무의미
- LLM-as-Judge의 평가 기준을 행동 계약에서 자동 생성할 수 있는가?
- 세션 간 신뢰 축적을 어떻게 정량화하는가?
